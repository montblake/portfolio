The parlor game Six Degrees of Kevin Bacon was inspired by the concept that any two people in the world could be linked by six personal connections or fewer. The game begins with some actor from a Hollywood film. The challenge is to link them — through films they have worked on and actors with whom they have worked — to Kevin Bacon with six degrees of separation or less.

The SIX_D_KB app offers two ways to start the game: with an actor chosen at random from the database or by the user entering a specific actor by name. And, if by chance you don't like Kevin Bacon, know who Kevin Bacon is, or for any other reason, you can change the game with a simple click.

The user can scroll through the actor's filmography, clicking on any title to reveal a cast list, and look for Kevin Bacon. Or they can select ANY actor from the cast lists in the hope THAT person has worked with Kevin Bacon. As you progress the app keeps track of who worked with whom, how many guesses remain, how much time remains of the clock.

Version 1.0

This was originally my first project at General Assembly in which the challenge was to build an app that gathered data from some free API and injected it into a simple html, css webpage using jQuery to handle dynamic rendering and user interaction

The real challenge turned out to be how much data I needed to deal with. In tackling "Six Degrees of Kevin Bacon", I needed to enter an actor's name and get back an official entry with name, image, and a list of all their films. Getting the filmography was easy. Filtering the list until only major Hollywood movies remained was a bit more interesting.

For major stars an initial list was often in the range of 500 or so entries; algorithm left us with 40 or 50 final films. The filmography list already contained not only those films' titles, but also the year they were released and poster images. What it didn't have was any kind of cast list. For that, the app needed to call a different API once for each film.

This first phase of the project was just about creating handling this information. I had to create some javascript objects to hold the data, the process had to be repeatable six times. Finally a visual frame had to be created to hold all this information as it arrived. There was no end, no real semblance of the game.

Version 2.0

For the August Mintbean Hackathon for Junior Devs, I revised this project. Now it has a Flask backend along with a React frontend. The main goal was to go further in "gamifying" the data. I added the game logic and highlighted the game aspects in visual layout.

Beyond that, I had another goal: I wanted to store the results of all our API calls and reduce the need for calls over time. Working with SQLAlchemy and a PostgreSQL database, I added film and actor tables and set up the various relationships. Finally I had to create the logic to get information from the database if available before making an api call and also to check the completeness of that information and if necessary make an api call to complete the information.

Unfortunately, the current state of the app has a big drawback. It is SLOW! If the info is in the database, everything works great. If not, the current version has to collect ALL of the information before it returns ANYTHING. All fifty movie casts must be individually looked up before you can even see the actor's picture. Ooops.

The first thing to take care of is to separate the various sub-processes of gathering actor data. The individual movie data needs to be an independent search in order to speed up the delivery of information. Perhaps it is possible to maintain two separate processes: one that prioritizes speed of deliverability, one that prioritizes building the database.

Additionally, I'd like to create a dashboard to track the size of the Database and its contents. Perhaps also track numbers of users, of searches, of API calls vs Database success. It would be nice if I could track the number of API calls and shut that down before we hit our daily maximum.

On a game level (visual level?) I can go further in adding highlights to various elements of the scoreboard and history that change over time. As the clock ticks down, does it get brighter or bigger or more central? The other option, I have realized, to laying everything out all over the space is to present only one element (or a selection of elements) at one time. A single focus playing space where the current focus is the only thing occupying your sight at that time. I suppose there is some sort of hybrid with a single focus playing area with buttons or images or text replacing one another at the focus -- while at the same time a dashboard is laid over a part of the screen.
