In the Eighties, he was a heart-throb. In the Nineties, a superstar. By the aughts... he was the subject of a drinking game.

Six Degrees of Kevin Bacon was a play on the concept that any two people in the world could be linked by six personal connections or fewer. Or something like that.

The game begins with some actor, any actor or actress. The challenge is to link them — through films they have worked on and actors with whom they have worked — with Kevin Bacon.

If we start with Brad Pitt. And Brad Pitt had worked with Kevin Bacon on some film. There would be one degree of separation between them. Easy, right?

But if Brad hadn't worked with Kevin but he HAD worked with Morgan Freeman who HAD worked with Kevin. That's two degrees of separation.

This project originated as a means to feature data gathered from a third-party api.

The actor info is easy enough to aquire, even if it takes two searches to get all the data we need. But each actor returns sometimes a few hundred results or more for their film output.

So the backend must filter those lists down to a list that balances completeness with the need for only truly legit movies. However, even this edited list, is rather large to search for every single actor.

I thought this iteration needed to focus three main goals: first to "gamefy" the data-centric app. This element has always been there but I wanted to spend time thinking through and building out further game features.

Secondly, I hoped to integrate a database into the search routine so that over time we might be able to relieve the reliance on api calls.

The last task was transitioning the app from an old-school jQuery tech stack to a modern, React frontend. Technology aside, I really wanted to think about components, modularity, and core engineering principles.

This project is built in Python with a Flask backend and deployed on Heroku. The database is Heroku's version of PostgreSql. The frontend is built in Node/Express and features a ReactJS UI.

In addition to handling the database, the backend handles api calls to third party sources, protecting its API keys, processing search results, provides json data, and serves up a static landing page.

I'd love to spend more time on the data delivery system and especially looking at the most efficient methods for handling the apps data needs.

The current system arose because it was the cleanest system I found at the time to make sure every possibility of incomplete data was tested, addressed, and remedied if need be. It was efficient for building up the database. But it is too slow in practice and causes a great overuse of resources.

Having seen a system operate with more complexity than on previous projects, I am more than ever aware of the value of unit testing and how the confidence in building up an app in discrete testable chunks could be incredibly beneficial.
